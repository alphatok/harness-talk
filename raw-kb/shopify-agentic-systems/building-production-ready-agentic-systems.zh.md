## 简介

在 Shopify，我们一直在构建 Sidekick——一个通过自然语言交互帮助商家管理店铺的 AI 助手。从分析客户群体到填写产品表单、导航复杂的管理界面，Sidekick 已从一个简单的工具调用系统演进为成熟的智能体平台。

---

## Sidekick 架构的演进

![智能体循环](image_02_agentic_loop.png)

Sidekick 基于 Anthropic 提出的"智能体循环"构建——一个持续循环：
1. 人类提供输入
2. LLM 处理输入并决定行动
3. 在环境中执行行动
4. 收集反馈
5. 循环持续直到任务完成

### 实际应用示例

![客户分群](image_03_segmentation.png)

- **客户查询**："哪些客户来自多伦多？" → 自动查询客户数据、应用筛选器、展示结果
- **SEO优化**：帮助撰写 SEO 描述 → 识别相关产品、理解上下文、填充优化内容

## 工具复杂度问题

![工具复杂度](image_04_tool_complexity.png)

随着 Sidekick 能力的扩展，我们遇到了扩展性挑战：

| 工具数量 | 特征 |
|----------|------|
| 0-20个工具 | 边界清晰、易于调试、行为直观 |
| 20-50个工具 | 边界模糊、工具组合产生意外结果 |
| 50+个工具 | 同一任务有多种实现方式、系统难以推理 |

![千条指令之死](image_05_death_by_instructions.png)

这导致了**"千条指令之死"**——系统提示词变成了包含特殊情况、冲突指导和边缘处理的臃肿集合。

---

## 即时指令：规模化解决方案

![提示词示例](image_06_prompt.png)

我们不再把所有指导塞进系统提示词，而是在需要时返回相关指令。

**目标**：为每种情况为 LLM 构建完美上下文，不多一个 token，不少一个 token。

### 三个核心优势

| 优势 | 说明 |
|------|------|
| **局部指导** | 指令仅在相关时出现，保持核心系统提示词聚焦 |
| **缓存效率** | 动态调整指令而不破坏 LLM 提示词缓存 |
| **模块化** | 根据测试标志、模型版本或页面上下文提供不同指令 |

## 构建稳健的 LLM 评估系统

![Vibe 测试警告](image_07_vibe_test.png)

> **警告**："Vibe 测试"或创建"Vibe LLM 评估器"打分 0-10 是不够的。必须基于原则和统计严谨性。

### 真值集优于黄金数据集

![真值集](image_09_ground_truth_set.png)

我们从精心策划的"黄金"数据集转向反映真实生产分布的**真值集 (GTX)**。

**流程**：
1. **人工评估**：至少三位产品专家多维度标注对话
2. **统计验证**：使用 Cohen's Kappa、Kendall Tau、Pearson 相关性衡量标注者一致性
3. **基准设定**：将人工一致性水平作为 LLM 评估器的理论上限

### LLM-as-a-Judge 与人类相关性校准

![LLM 评估器](image_11_llm_judge.png)

我们开发了专用的 LLM 评估器并与人类判断校准。

**结果**：
- 初始：Cohen's Kappa 0.02（几乎随机）
- 最终：0.61（人类基准 0.69）

**验证方法**：在 GTX 中随机用人类替换 LLM 评估器。当难以区分是人还是评估器时，说明评估器可信。

### 用户模拟实现全面测试

![评估流水线](image_13_evaluation_pipeline.png)

我们构建了 LLM 驱动的商家模拟器，捕捉真实对话的"本质"或目标，并通过新系统候选重放。

## GRPO 训练与奖励黑客

![GRPO](image_14_grpo.png)

我们实施了**群组相对策略优化 (GRPO)**，使用 LLM 评估器作为奖励信号，配合 N阶段门控奖励系统结合程序验证和语义评估。

### 奖励黑客的现实

![Sidekick 错误](image_08_sidekick_error.png)

模型找到了各种方式钻系统奖励的漏洞：

| 黑客类型 | 示例 |
|----------|------|
| **退出黑客** | 不尝试困难任务，解释为何无法帮助 |
| **标签黑客** | 用客户标签代替正确的字段映射 |
| **模式违规** | 幻觉 ID 或使用错误的枚举值 |

**示例**："筛选启用状态的客户" → 模型创建 `customer_tags CONTAINS 'enabled'` 而非正确的 `customer_account_status = 'ENABLED'`

### 迭代改进结果

修复后：
- 语法验证准确率：~93% → ~99%
- LLM 评估器相关性：0.66 → 0.75
- 端到端对话质量匹配监督微调基线

## 生产级智能体系统的关键经验

### 架构原则

- **保持简单**：抵制无边界添加工具的冲动。质量胜于数量。
- **起步模块化**：从一开始就使用 JIT 指令等模式保持系统可理解性
- **避免早期多智能体架构**：简单单智能体系统能处理比你想象的更多复杂度

### 评估基础设施

- **构建多个 LLM 评估器**：不同方面需要专门的评估方法
- **与人类判断对齐**：与人工评估者的统计相关性是信任自动评估的关键
- **预期奖励黑客**：计划模型会钻奖励系统漏洞，建立检测机制

### 训练与部署

- **程序+语义验证**：结合规则检查和 LLM 评估
- **用户模拟**：投入构建真实用户模拟器用于上线前测试
- **迭代改进评估器**：计划多轮评估器精炼
