# TLDR: 扩展托管智能体——大脑与双手解耦

> 核心：通过接口抽象实现组件解耦，让实现可以自由演进而接口保持稳定

---

## 一、核心定义

**一句话**：解决 harness 编码的模型假设会随时间过时的问题，通过将智能体虚拟化为 Session/Harness/Sandbox 三个接口，使组件可独立替换、扩展和恢复。

---

## 二、架构分层

| 层级 | 组件 | 本质 | 核心细节 |
|------|------|------|----------|
| **Session** | 事件日志 | **追加式持久存储** | 1. `getEvents()` 可按位置切片查询；2. 独立于 Claude 上下文窗口，支持不可逆压缩后的回溯 |
| **Harness** | 控制循环 | **大脑（无状态）** | 1. 通过 `wake(sessionId)` 从任意失败恢复；2. 只路由调用，不持有凭证 |
| **Sandbox** | 执行环境 | **双手（可替换）** | 1. 通过 `execute(name, input) → string` 作为工具调用；2. 容器死亡 → harness 捕获错误 → Claude 决定重试 → 新容器 `provision()` |

---

## 三、避坑指南

| 陷阱 | 教训 |
|------|------|
| **宠物容器** | 所有组件放一个容器 = 容器失败时 session 丢失、调试困难、凭证暴露 |
| **上下文焦虑** | Sonnet 4.5 会在上下文满前提前结束 → 加重置；Opus 4.5 没这行为 → 重置变死代码 |
| **凭证同沙箱** | 提示词注入只需让 Claude 读环境变量 → 结构修复：凭证在 vault 或绑定资源 |
| **假设 baked in** | Harness 假设资源在旁边 → 客户 VPC 必须对接网络 → 假设变成限制 |

---

## 四、落地参考（Agent 系统设计）

### 1. 接口设计原则

```
Session: wake(id), getSession(id), emitEvent(id, event), getEvents()
Sandbox: execute(name, input) → string, provision({resources})
Harness: 无状态，可从任意点恢复
```

**落地要点**：接口对形态有立场，对实现无立场。`execute()` 不知道沙箱是容器还是手机。

### 2. 凭证隔离模式

| 模式 | 实现 | 适用场景 |
|------|------|----------|
| **绑定资源** | Git: 初始化时注入 token 到 remote | 资源有天然凭证机制 |
| **Vault + Proxy** | MCP: 代理从 vault 取 token 调外部服务 | 自定义工具、OAuth |

**落地要点**：Harness 永不接触凭证，沙箱生成代码永不能读凭证。

### 3. TTFT 优化策略

| 问题 | 解耦后效果 |
|------|-----------|
| 每个大脑一个容器 | 容器只在需要时 `execute()` 创建 |
| 全量初始化 | 推理从拉取 session 事件开始，不等容器 |

**落地要点**：p50 TTFT -60%，p95 TTFT -90%。按需延迟加载是关键。
