# 微信小程序 AI SKILL 最佳实践 — TLDR

> 来源：[微信小程序 AI 最佳实践](https://developers.weixin.qq.com/miniprogram/dev/ai/best-practices.html)

---

## 一、产品设计核心原则

- **模糊意图承接**：AI 理解非结构化需求（"想喝点清爽的"），引导至原子组件完成确认和下单
- **对话式修改**：用户可直接对话修改规格、时间、地点，无需跳出对话
- **历史数据可操作**：支持"再来一杯上次喝的"等高频效率诉求
- **动态更新**：订单/服务组件随状态实时更新，不静态展示

---

## 二、信息源权重层级

| 权重 | 信息源 | 作用 |
|------|--------|------|
| ★★★★★ | 接口返回 content | 事实 + 动作 |
| ★★★★ | mcp.json description | 选接口 |
| ★★★★ | inputSchema.description | 填参数 |
| ★★★ | SKILL.md | 业务流程编排 |

**常见错位**（核心陷阱）：
1. ✗ 接口功能写进 SKILL.md → 应写进 mcp.json description
2. ✗ 跨接口规则写进单接口 description → 应写进 SKILL.md
3. ✗ 功能描述写进 content → content 只承载本次结果与下一步

---

## 三、通用写作铁律

1. **同约束只写一处**，重复易冲突
2. **硬约束放权重高的位置**，能代码校验优先代码校验
3. **每条禁令配出口**：不写"不要做 X"而不给替代动作

---

## 四、mcp.json 声明规范

### 接口 description
- 首句声明**业务对象**而非入参
- 语义化接口名（`searchDrinks` 优于 `search`）
- 不写内部实现细节
- 同名字段统一命名（始终 `drinkId`，不混用 `itemId`）

### 字段 description
- **普通字段**：多举例 + 配缺省处理（单一举例易被当默认值）
- **ID 字段**：显式声明取值来源接口（防模型构造凑值）

---

## 五、content 返回规范

- **"事实 + 动作"两段式**：先陈述客观状态，再给下一步
- **失败/空结果三要素**：陈述原因 → 给出出口 → 指出不应做（配对联出）
- 渲染用数据走 `_meta`，AI 理解内容走 `content`

---

## 六、SKILL.md 写作重点

- 节点名与 mcp.json 完全一致
- 从用户意图分支，每分支对应典型表达
- 标注用户操作节点 vs AI 决策节点
- 依赖关系表按接口维度列前置条件
- 业务约束（输出形态、执行顺序、并发串行、数据来源）

---

## 七、上行消息文案原则

- **用户视角**：第一人称、生活化口语
- **不可上行系统消息**：异常也要转换成用户操作表达
- **仅限已明确表达的信息**，不预判不补充
- **敏感信息脱敏**，指代不模糊
