## 清晰度胜过"更努力思考"

上个月，Anthropic 发布了一项让我深受触动的发现。增加反思步骤——听起来像是"更努力地思考"——实际上降低了他们 AI 的准确性。这清楚地表明，更好的路径是简化 LLM 推理，而不是强行增加"思考"。这是一种如果你一直在用 LLM 构建产品就会觉得显而易见的结论，但看到它被明确写出来，确实是一记警钟。

七月的大部分时间里，我都在观察模型在自己内部的独白中绊倒。我在 prompt 中叠加的"逐步推理"或"自我反思"指令越多，输出的偏移就越严重。这呼应了我在生产环境中几乎每周都能看到的模式——追求更深层的推理只会叠加噪音。如果你正试图[交付可靠的功能](https://codewithcaptain.com/build-reliable-llm-pipelines/)，现在正是退后一步问问自己：你是把事情变好了，还是只是把它们变复杂了。

![Simplify LLM reasoning: 一团乱麻的绳子旁边是一条通向目标的清晰笔直路径](images/simplify-llm-reasoning/simplify-llm-reasoning-c74d0631.jpg)

直说吧。问题的根源就是 prompt 臃肿，简单明了。过载和模糊的 prompt 让模型迷失，不可靠的输出最终进入你的产品。我发布过看起来很聪明的 prompt，但实际效果更差——因为它们试图做的事情太多了。

问题在哪里。当你的真实目标被埋没在细节中时，模型并不会变得更好——它只是放大混乱。我见过充满离题、误解和随机脑暴的输出，与我们的实际需求毫无关系。如果模型的"推理"步骤因为 prompt 充满了不确定性而被拉长，那么这种不确定性就会直接泄漏到答案中。讽刺的是，更多的所谓"思考"只是通过重新混合不相关的上下文来叠加错误，而不是深入相关之处。

这就是核心论点。清晰度带来结果，过度复杂化每次都会适得其反。你应该[让意图明确](https://codewithcaptain.com/write-clearly-with-ai/)，保持内部步骤最小化——刚好够完成任务就行——并始终用你和系统的外部检查来锚定结果。

是时候重新思考我们如何设计 LLM 功能了，尤其是现在 Anthropic 的研究让这个问题的利害关系如此清晰。

## 为什么 LLM 在重新混合模式（而非推理）

去掉炒作。大型语言模型在任何真正意义上都不是"思考"——它们只是非常擅长重新混合模式。实际发生的事情是下一个词预测。这些系统通过自监督预训练从海量文本中捕捉模式（[cset](https://cset.georgetown.edu/article/the-surprising-power-of-next-word-prediction-large-language-models-explained-part-1/)）。你不是在指示一个推理机器；你是在提示一个超级驱动的自动补全。这重新定义了所有事情。如果你期望逻辑，你只能靠运气。但如果你给它清晰的模式，它就能精确地按设计工作。

这是我经常遇到的情况。当我埋没问题或在 prompt 中堆叠额外、不相关的信息时，模型就会漂移。prompt 开头或结尾的相关细节会被捕捉到，但夹在中间的任何内容几乎立即丢失（[tacl-mit](https://direct.mit.edu/tacl/article/doi/10.1162/tacl_a_00638/119630/Lost-in-the-Middle-How-Language-Models-Use-Long)）。这就像用极小的字体低语你的真实问题。我不断看到听起来很精炼但完全偏离重点的输出，都是因为我在 prompt 中填充了多余信息，希望它能"想得更好"。

你可能会认为通过逐步推理或自我批评来减少 LLM 过度思考的努力会有帮助。但每次我投入更多的上下文或额外步骤，输出就变得更模糊、更不相关。

我向它们投入的上下文、步骤或"反思"越多，得到尖锐有用答案的可能性就越低。不是改进，实际发生的是模型通过重新混合误解来堆积噪音。我试过那些思维链 prompt 和所有的反馈循环。事实证明，对于大多数任务，带反馈的自我纠正并不有效，除非该任务真的适合这种方法（[tacl-feedback](https://direct.mit.edu/tacl/article/doi/10.1162/tacl_a_00713/125177/When-Can-LLMs-Actually-Correct-Their-Own-Mistakes)）。大多数时候，你只是在让模型生成更多的文本，而不是更多的清晰度。所以复杂性实际上会叠加错误；开始时是对细微差别的追求，最终却得到更少的相关性。

有一次，我花了一整个下午调整一个财务摘要工具的 prompt，坚信再多一个推理步骤就能帮助模型捕捉到一个顽固的 off-by-one 计算错误。结果它开始幻觉出文档中从未存在过的全新数据类别。老实说，到某个时候我都不确定自己在解决什么问题了。那个经历依然有点刺痛——但它可能是我得到过的最直接的教训。

我很快意识到，在模型中隔离推理只会让它绕圈子。要求它"与自己争论"或在没有新事实的情况下进行内部反思，会把 prompt 变成一个回音室，强化它开始的任何偏见或混乱。你会陷入自信的胡言乱语——只是一个反馈循环中堆积的噪音层。

所以真正的教训是：当你简化 LLM 推理时，清晰度胜过了复杂性。保持 prompt 精简，让模型聚焦，并用外部的反馈来锚定输出。这使可靠的输出成为可能——而且它实际上加快了迭代速度。

## 可构建的原则：将清晰度转化为可靠的 LLM 组件

从明确的意图提示开始。让它不可错过。对我来说，输出质量的最大飞跃来自我把 prompt 写得像一份合同：目标、边界、约束条件，以及什么算成功，都用清晰的语言写出来。精确规范的清晰度驱动结果，而非运气。当我开始提前定义我想要的东西时，整个过程就顺畅了。[明确框架减少了来回沟通](https://www.anthropic.com/news/system-prompts)，所以与其无休止地调整，你更快就能获得可以信任的输出。把 LLM 当做一个新队友。明确说明"完成"应该是什么样子，给它一个名字。

接下来，只保留实际需要的推理步骤。把所有可能的步骤都扔进去，或者通过填充逻辑来"彻底"，这很诱人。但现实点。这不是聪明竞赛，你也不需要把所有东西都扔进去。把关每个步骤。如果任务不需要，就去掉它。

每个可靠的系统都需要事实锚点。追求 LLM 的[外部校验](https://codewithcaptain.com/build-resilient-llm-systems/)——通过检索获取事实、通过工具调用获取行动、多个评估者，或直接的人工反馈。新鲜输入打破了循环，让你的系统保持脚踏实地，而不是陷入自己的假设中。

六个月前，我对待 prompt 就像炖汤——把所有调料都扔进去希望变出魔法。这种方法只是掩盖了模型不知道的东西。现在，我边尝边调，一次只加一种东西。更好的是，我会寻求第二意见——同事的检查、检索结果、真实世界的验证器。

[将这些原则直接映射到你的架构上](https://codewithcaptain.com/how-to-decouple-ai-architecture/)，以下是有效的方法。意图序列化器为模型编写契约，捕获目标和限制。步骤限制器执行你的推理预算，防止失控的"思考"。验证器连接到外部现实——检索、工具使用或提供真正外部视角的评估者。引导式脚手架使功能开发更简单、更可靠，因为每一步都有自己的目的，并附有外部事实的锚点。

如果你保持规范清晰、步骤精简、校验外部化，LLM 功能就会以你真正想要的方式运行——可靠、快速、可问责。

## 何时增加步骤（何时保持精简）

> 很容易想象，如果一点推理是好的，更多一定更好——但我不断看到相反的情况。在某些情况下，思维链、自我批评甚至重型脚手架确实能带来真正的改变。如果你要规划五个截止日期、修补工具链中的缺口、或将输出适配到严格的数据模式，你需要那些明确的步骤。
>
> 模型可以从"出声思考"中受益，因为有真正的模糊性或多个约束需要处理。但在标准摘要、问答或任何与事实检索相关的任务中，堆叠内部步骤只会适得其反。这些额外的"反思"不是提高准确性，而是固化错误、制造自信的胡言乱语，有时甚至失控。你要求的元推理越多，模型就越偏离——尤其是当你的实际目标一直都很简单的时候。所以，更多的思维链或自我批评不是魔法。它们可能固化错误或产生听起来合理的胡言乱语。关键在于将步骤与实际复杂性匹配，而不是默认添加。

我希望能为每个模型提供一个通用规则，但现实更复杂。同样精心构造的 prompt 在一个 LLM 上完美运行，在另一个上可能彻底失败——甚至是同一供应商的不同模型。我曾惊讶于一个 prompt 在一个模型上表现良好，在另一个模型上却完全失效。你需要跨模型验证，不要假设你的聪明思维链或脚手架能通用。

如果你想看到什么真正有效，构建一个快速的 LLM Agent 输出验证协议。从单次基线开始。然后叠加一个步骤门控版本（显式推理或自我批评，刚好能看到是否有效果）。最后，添加一个外部校验变体——在模型自身循环之外锚定或验证输出的东西。离线评估这些版本，使用保留数据或黄金数据集，然后在任何面向用户的内容中推出。每天，我都依赖回归测试来防止聪明的想法将 bug 潜入生产环境，这种方法比我任何头脑风暴都更多地捕捉到了隐藏的漂移。

还有一件事。[人工反馈是粘合剂](https://codewithcaptain.com/how-design-ai-human-workflows/)。它明确了你想要什么，标记模型偏离轨道的地方，并闭合你的代码、你的 prompt 和你的评估者之间的循环。这就是你获得可靠、可构建的结果的方式——那种你真正可以发布的结果。

## 让清晰成为默认：务实的路线图

这里有一个实用的计划来简化你的 LLM prompt，从今天开始就可以用。首先定义你的意图——你真正想让模型做什么，用简单的语言说出来。接下来，限制模型可以走的步骤数量；不要让它游荡或进入元层面，除非你能指出具体需求。然后用某种外部检查来锚定结果——检索的事实、现有的 API 或简单的同行评审。离线测量这一切的效果，使用保留数据，然后投入生产。每一轮，用你学到的东西来收紧系统。我们已经让 prompt 臃肿成为默认太久了——让我们翻转它，让最小化、光照良好的路径成为团队首选的方案。

人们担心这会花太长时间，但事实是它减少了你在试错上浪费的时间。周期加快，输出稳定。我只是通过抛弃旧的 prompt 垃圾就节省了几个小时——我一点也不怀念那些救火的日子。

另一个误区是，当你减少步骤时会失去细微差别。实际上，用清晰的 prompt 引导模型会保持对重要变量的关注，所以你在真正重要的地方保持了细微差别。你总是可以在真正需要边缘情况时使用引导式脚手架。有时，单次干净的传递就赢了。其他时候，你需要几条轨道——不管怎样，你跳过了无尽内部独白的混乱。

关于泛化——模型是不同的，这一点绕不开。所以，我把 prompt 视为规范驱动的接口，而不是为某个 LLM 硬编码的脚本。构建一个小型测试套件，在多个模型上运行，并选择稳定的评估器来发现哪里发生了漂移。我预期 prompt 在不同供应商之间表现不同，但当你提前规范并在模型外部锁定检查时，[你仍然能得到可靠的功能](https://codewithcaptain.com/build-reliable-llm-pipelines/)。

你已经看到了 prompt 臃肿的陷阱。这周在你的一个已有功能上尝试一个清晰度优先的实验。你可能会对可靠输出变得多么容易感到惊讶。

使用清晰的目标、约束和语气生成 AI 驱动的草稿，然后用外部检查在同一个地方迭代，这样你就可以快速生成可靠的内容，而无需 prompt 臃肿。
