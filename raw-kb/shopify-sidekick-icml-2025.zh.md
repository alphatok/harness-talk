## 一、架构

### 核心设计

- **标准 Agent 循环**：LLM + 工具集 → 与环境交互
- **无显式工作流**：模型自行决定如何处理任务
- **不受约束**：行为不被刚性限制；模型能从工具错误中自行恢复
- **关键策略**：底层原语工具（非任务特定函数）

### 初始核心技能 (V1)

| 技能 | 说明 |
|------|------|
| 客户细分 | Text-to-SQL 创建客户列表（营销活动、折扣码） |
| 分析 | Text-to-SQL 查询销售/业务数据 |
| 导航 | 在 1000+ 后台页面中引导商家 |
| 表单填写 | 创建/更新 Shopify 任意资源（产品、折扣、SEO） |
| 帮助 | 基于 RAG 的 Shopify 帮助中心问答 |

### 教训：工具膨胀

- 工具数量自然增长到 **20-50+** → 边界模糊
- **"千条指令致死"**：系统提示变得无法评估，指令冲突
- 工具越多，回复质量越低

### 解决方案：即时指令 (Just-in-Time Instructions)

将指令与**工具结果一同返回**，而非全部放在系统提示中。

| 优势 | 说明 |
|------|------|
| 精简系统提示 | 核心行为（个性、长度、国际化）留在提示中；边缘情况移到工具响应 |
| 缓存友好 | 工具指令位于对话历史底部 → 只需重建 n-1 轮缓存 |
| 模块化 / 低影响范围 | 修改工具指令只影响该工具，不影响整个系统 |

### Agent 交接（探索中）

- 主 Agent 将任务委托给专门的子 Agent（各有独立提示 + 工具）
- 子 Agent 完成任务后交回主 Agent 回复用户
- **注意**：聊天系统有延迟顾虑 — 仅适用于后台任务

### 架构要点

1. **尽可能保持简单** — 一个简单的 Agent 循环就能创造巨大价值
2. **不要跳入多 Agent** — 大多数场景不需要
3. **质量 > 数量** — 一组高质量底层原语胜过 50 个零散工具
4. **保持模块化** — 用 JIT 指令等模式减小影响范围

## 二、LLM 评估

### 从"凭感觉测"到系统化评估

- 原始方法：凭感觉测试 → 发布 → 出错了回滚（不可持续）
- 无法判断改动是否有效；系统提示难以调试

### 评估架构

```
用户模拟器 (LM 驱动) → 候选系统 → LM 评判员
```

- **用户模拟器**：LM 驱动的商家模拟器，复现真实对话意图
- **候选系统**：新提示 / 新工具 / 新组件（一次只改一个）
- **LM 评判员**：阅读完整对话，按多维度标准评分

### "真实情况"集 (Ground Truth Set)

- ~1000+ 条对话（持续增长）
- 遵循**生产分布**
- 产品经理定义 **5-7 个评估维度**：
  - 目标完成度
  - 安全性
  - 商家情绪
  - 事实依据
  - 等
- 3-5 位人类专家（PM + 团队成员）为每条对话标记评分
- 统计一致性衡量（Kendall's Tau / Cohen's Kappa，目标 ~0.7）
- **这就是规格说明书** — 不是静态的黄金集

### 基于回放的仿真

- 收集真实生产对话 → 提取用户意图与目标
- LM 用**候选系统**重放该对话（而非生产系统）
- **AA 测试**：相同系统 + 相同对话 → 验证 LM 评判员与生产评分一致
- 验证通过后，即可用于 AB 对比

### LM 评判员迭代

- 版本 0 → 相关性 0.02；迭代到 0.11 → 持续提升
- 评判员**不是逐步训练的判别器**（不同于 GAN）— 从一开始就尽可能强
- 持续飞轮：新边缘案例 → 加入真实情况集 → 重新提示工程评判员

### 关键原则

> 离线评估做越多，在线 AB 测试前就能尝试越多的实验、超参数和提示版本。

## 三、GRPO / 强化学习训练（概览）

- 利用商家模拟器 + LM 评判员构建 RLHF 流程
- 真实情况集随生产分布持续演进 → 评判员提示不断调整
- 强化学习同时改进模型和评估流程

---

## 总结

| 领域 | 核心洞察 |
|------|---------|
| 架构 | 底层工具 + JIT 指令 > 系统提示臃肿 |
| 工具 | 质量胜于数量；避免"千条指令致死" |
| 评估 | LM 用户模拟器 + LM 评判员 > 凭感觉测试 |
| 数据 | 真实情况集 = 活着的规格；随生产边缘案例持续增长 |
| 强化学习 | 评判员从一开始就强；随真实情况漂移持续演进 |
