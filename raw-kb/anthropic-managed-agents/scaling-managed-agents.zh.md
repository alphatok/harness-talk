# 扩展托管智能体：将大脑与双手解耦

**来源**: https://www.anthropic.com/engineering/managed-agents  
**发布日期**: 2026年4月8日  
**作者**: Lance Martin, Gabe Cemaj, Michael Cohen

---

![架构图](diagram_01.png)

Harness（控制框架）编码了关于模型能力的假设，这些假设随着模型改进会过时。Managed Agents——我们为长时间运行智能体提供的托管服务——构建在稳定的接口之上，这些接口在 harness 变化时保持不变。

---

## 引言

工程博客的一个持续主题是如何构建有效的智能体并为长时间运行设计 harness。这些工作的共同线索是：harness 编码了关于 Claude 无法独立完成任务的假设。但这些假设需要经常质疑，因为它们会随着模型改进而过时。

一个例子：在之前的工作中，我们发现 Claude Sonnet 4.5 在感知到上下文限制临近时会提前结束任务——这种行为被称为"上下文焦虑"。我们通过在 harness 中添加上下文重置来解决。但当我们在 Claude Opus 4.5 上使用同一个 harness 时，这种行为消失了。重置变成了死代码。

我们预期 harness 会持续演进。因此我们构建了 Managed Agents：Claude Platform 上的托管服务，通过一组小型接口为您运行长时间运行的智能体——这些接口设计用于超越任何特定实现的生命周期。

---

## 操作系统类比：为"尚未构思的程序"设计系统

构建 Managed Agents 意味着解决计算机领域的一个老问题：如何为"尚未构思的程序"设计系统。几十年前，操作系统通过将硬件虚拟化为抽象——进程、文件——解决了这个问题。这些抽象足够通用，能支持当时还不存在的程序。抽象的生命周期超越了硬件。`read()` 命令不关心它访问的是1970年代的磁盘还是现代 SSD。顶层抽象保持稳定，底层实现自由变化。

Managed Agents 遵循同样的模式。我们将智能体的组件虚拟化：

| 组件 | 角色 |
|------|------|
| **Session（会话）** | 所有发生事件的追加式日志 |
| **Harness（控制框架）** | 调用 Claude 并将 Claude 的工具调用路由到相关基础设施的循环 |
| **Sandbox（沙箱）** | Claude 可以运行代码和编辑文件的执行环境 |

这允许每个组件的实现被替换而不干扰其他组件。我们对这些接口的形态有立场，但对接口背后的实现没有。

## 不要养宠物

![耦合设计](diagram_02.png)

我们最初将所有智能体组件放在单个容器中——session、harness 和 sandbox 共享同一个环境。这种方法的优点是：文件编辑是直接的系统调用，无需设计服务边界。

但将所有东西耦合到一个容器中，我们遇到了一个老基础设施问题：我们养了一只**宠物**。在 pets-vs-cattle（宠物 vs 牛群）类比中，宠物是有名字、精心照料、不能丢失的个体；牛群是可互换的。在我们的案例中，服务器变成了那只宠物；如果容器失败，session 就丢失了。如果容器无响应，我们必须修复它。

### 耦合设计的问题

| 问题 | 描述 |
|------|------|
| **调试卡住的 session** | WebSocket 事件流无法告诉我们失败发生在哪里。harness bug、丢包、容器离线——表现都一样。 |
| **安全边界** | Claude 生成的不可信代码与凭证在同一个容器运行——提示词注入只需让 Claude 读取自己的环境。 |
| **网络对接** | 客户必须将他们的网络与我们对接才能让 Claude 访问他们的 VPC，或者在他们自己的环境中运行我们的 harness。 |

## 将大脑与双手解耦

![解耦设计](diagram_03.png)

解决方案是将**"大脑"**（Claude 和它的 harness）与**"双手"**（执行操作的沙箱和工具）以及**"session"**（事件日志）解耦。

### 核心接口

| 接口 | 功能 | 目的 |
|------|------|------|
| `execute(name, input) → string` | 调用沙箱的工具 | 大脑像调用其他工具一样调用双手 |
| `wake(sessionId)` | 重启 harness | Harness 变成牛群 |
| `getSession(id)` | 获取事件日志 | Session 在 harness 外部 |
| `emitEvent(id, event)` | 写入 session | 智能体循环中保持持久记录 |
| `provision({resources})` | 创建沙箱 | 用标准配方重新初始化容器 |
| `getEvents()` | 读取 session 事件 | Claude 上下文窗口外的上下文对象 |

### Harness 离开容器

Harness 不再住在容器里。它像调用其他工具一样调用容器：`execute(name, input) → string`。容器变成了**牛群**。如果容器死亡，harness 捕获失败作为工具调用错误并传回 Claude。如果 Claude 决定重试，可以用标准配方 `provision({resources})` 重新初始化新容器。我们不再需要修复失败的容器。

### 从 harness 失败中恢复

Harness 也变成了牛群。因为 session 日志在 harness 外部，harness 中任何东西都不需要在崩溃中存活。当一个失败时，可以用 `wake(sessionId)` 重启新的，用 `getSession(id)` 获取事件日志，从最后一个事件恢复。

![恢复流程](diagram_04.png)

### 安全边界

在耦合设计中，Claude 生成的不可信代码与凭证在同一容器运行——提示词注入只需让 Claude 读取自己的环境。一旦攻击者获得这些 token，他们可以生成新的无限制 session 并委托工作给他们。

结构性修复是确保 token 从 Claude 生成代码运行的沙箱中永远不可达。

| 模式 | 实现 |
|------|------|
| **凭证绑定资源** | Git：在沙箱初始化时用访问 token 克隆仓库，接入本地 git remote。Push/pull 在沙箱内工作，智能体从不处理 token。 |
| **凭证在 vault** | MCP 工具：OAuth token 存储在安全 vault。Claude 通过专用代理调用 MCP 工具，代理从 vault 获取凭证。 |

## Session 不是 Claude 的上下文窗口

长时间运行任务通常超出 Claude 上下文窗口长度，标准解决方法都涉及关于保留什么内容的不可逆决定：

| 技术 | 描述 |
|------|------|
| **压缩（Compaction）** | Claude 保存上下文窗口的摘要 |
| **Memory 工具** | Claude 将上下文写入文件 |
| **上下文裁剪** | 选择性删除 token（旧工具结果、thinking blocks） |

但选择性保留或丢弃上下文的不可逆决定可能导致失败。很难知道未来回合需要哪些 token。

在 Managed Agents 中，session 提供一个在 Claude 上下文窗口之外的上下文对象。接口 `getEvents()` 允许大脑通过选择事件流的位置切片来查询上下文：

- 从上次停止读取的位置继续
- 回退到特定时刻前几个事件
- 在特定操作前重新读取上下文

我们将**可恢复上下文存储**（session）和**任意上下文管理**（harness）的关注点分离，因为我们无法预测未来模型需要什么样的特定上下文工程。

## 多大脑、多双手

### 多大脑

将大脑与双手解耦解决了我们最早的客户抱怨。当团队想让 Claude 访问他们自己 VPC 中的资源时，唯一的路径是将他们的网络与我们对接。

**性能收益**：最初将大脑放在容器中意味着多个大脑需要多个容器。每个 session 都要支付完整的容器设置成本——即使那些永远不会触碰沙箱的 session。

**首 token 时间（TTFT）**测量 session 在接受工作和产生第一个响应 token 之间等待多久。这是用户最敏感感知的延迟。

| 指标 | 改进 |
|------|------|
| **p50 TTFT** | 降低 ~60% |
| **p95 TTFT** | 降低 ~90% |

解耦意味着容器只在需要时由大脑通过工具调用创建。推理可以在编排层从 session 日志拉取待处理事件后立即开始。

### 多双手

我们还希望每个大脑能连接多个双手。实践中这意味着 Claude 必须推理多个执行环境并决定向哪里发送工作——这是比在单个 shell 中操作更难的认知任务。

随着智能提升，单个容器变成了限制：当那个容器失败时，我们丢失了大脑正在操作的所有双手的状态。

解耦将每个双手变成一个工具：`execute(name, input) → string`。该接口支持：

- 任何自定义工具
- 任何 MCP server
- 我们自己的工具

Harness 不知道沙箱是容器、手机还是 Pokémon 模拟器。而且因为没有双手与任何大脑耦合，**大脑之间可以传递双手**。

## 总结

我们面临的挑战是一个老问题：如何为"尚未构思的程序"设计系统。操作系统通过将硬件虚拟化为足够通用的抽象已经存活了几十年。

Managed Agents 是同样精神的**元 harness**，对 Claude 未来需要的特定 harness 没有预设立场。它是具有通用接口的系统，允许多种不同的 harness。

**元 harness 设计意味着对 Claude 周围的接口有立场**：

- Claude 需要操作状态的能力（session）
- Claude 要执行计算的能力（sandbox）
- Claude 需要扩展到多大脑和多双手的能力

我们设计接口使其可以在长时间内可靠安全地运行。但我们不对 Claude 需要的大脑或双手的数量和位置做假设。
