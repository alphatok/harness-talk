# Lessons from building Claude Code Skills: 关键发现与最佳实践建议

## 1. 核心定义

**Skills 不是 markdown 文件，而是包含指令、脚本和资源的文件夹**。代理可发现并利用它们更准确高效地完成任务。核心痛点在于：大规模场景（数百个 skills）下，如何设计、组织和分发 skills 才能最大化开发效率。

## 2. 架构/方法分解

### 【Skill 九分类框架】
**本质**：按功能将 skills 分为 9 类，单一职责原则，跨类别会显著降低代理准确性。
- 分类维度：从"知识参考"（库/API 使用）到"自动化操作"（CI/CD、基础设施运维）逐层递进
- 关键洞察：验证类（Product Verification）对输出质量影响最大，值得投入最多资源

### 【Gotchas 陷阱机制】
**本质**：最高信号价值的内容单元，从实际失败点构建，持续更新迭代。
- 常见陷阱示例："订阅表是只追加的，要找版本最高的行，不是最新创建的行"
- 实践要点：不能一次性写好，必须随时间不断积累

### 【渐进式披露】
**本质**：利用文件系统层次组织 skill 内容，代理按需加载而非一次性全量读取。
- 文件结构：`SKILL.md`（入口）+ `references/`（详细文档）+ `assets/`（模板）+ `scripts/`（脚本）
- 优势：减少 Token 消耗，避免上下文污染

### 【按需钩子机制】
**本质**：仅在特定 skill 被调用时激活的安全防护钩子，生命周期限于会话期间。
- `/careful`：阻止 `rm -rf`、`DROP TABLE`、`force-push` 等危险操作
- `/freeze`：限制编辑范围到特定目录

### 【描述为模型而写】
**本质**：description 是代理扫描后决定"是否触发此 skill"的依据，非人类文档摘要。
- 应包含触发词（如"babysit"）而非泛泛的功能描述
- 决定模型在数百个 skill 中选择哪一个的关键信息

## 3. 避坑指南/反直觉教训

| # | 错误认知 | 实际教训 |
|---|---------|---------|
| 1 | Skills 只是 markdown 文件 | 是文件夹，可包含脚本/资源/数据/配置 |
| 2 | 描述写给人类看 | 描述是给模型的触发器，需包含触发词 |
| 3 | 写完整使用说明 | 只写非显而易见的信息（边缘情况、陷阱、与默认行为差异） |
| 4 | 给出精确步骤指令 | 保留灵活性，过度约束限制适用场景 |
| 5 | 一次性写好不再维护 | Gotchas 部分需持续更新，接收新失败点 |
| 6 | 技能越多越好 | 每个检入仓库的 skill 都增加模型上下文负担 |

## 4. 我的具体关注点：Agent 系统设计与实现落地

### 细节 1：Skill 分类框架指导工具库设计
**实操方法**：设计 Agent 的工具注册表时，按 9 类分组。为每类设置不同调用策略——验证类工具优先调用，数据获取类按需调用，运维类设置安全确认（Human-in-the-loop）。

### 细节 2：Gotchas 机制融入工具执行反馈循环
**实操方法**：工具调用失败时自动将错误信息追加到该工具的"陷阱数据库"。下次调用前检索历史陷阱并注入提示词，实现自动学习与避坑。这与我们之前讨论的 function call 失败重试机制天然互补。

### 细节 3：渐进式披露优化工具上下文管理
**实操方法**：将工具定义拆分为"基础描述"（必传）和"详细文档"（按需加载）。模型首次调用时只传基础描述，需要更多信息时通过再次查询加载详细文档。显著减少 Token 消耗，避免每轮都支付完整工具 schema 的输入 token 成本。
