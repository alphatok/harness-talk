# 生产级智能体系统最佳实践

> 基于 Shopify Sidekick 生产经验提炼的核心落地要点

---

## 一、架构设计

### 1.1 工具边界管理

| 阶段 | 工具数 | 行动 |
|------|--------|------|
| 初期 | ≤20 | 明确边界、单一职责 |
| 中期 | 20-50 | 定期审查重叠、合并同类 |
| 后期 | 50+ | 分层抽象、JIT 动态加载 |

**关键**：工具数量不是能力指标，边界清晰才是。

### 1.2 JIT 即时指令

**不要**：把所有指导塞进系统提示词 → "千条指令之死"

**要做**：
- 按场景动态返回相关指令
- 保持核心系统提示词聚焦基础行为
- 利用缓存效率 + 版本控制

**目标**：每种场景的上下文精确匹配，不多不少。

### 1.3 单智能体优先

- 单智能体可处理比你想象的更多复杂度
- 多智能体架构增加调试难度、通信成本
- 只有明确需要并行协作时才引入

## 二、评估体系

### 2.1 禁止 Vibe 测试

**错误做法**：
- ❌ "打分 0-10"
- ❌ "感觉还行"
- ❌ 只看少数样本

**正确做法**：
- ✅ Cohen's Kappa ≥ 0.6
- ✅ ≥3人标注、统计验证一致性
- ✅ 人类一致性作为 LLM 评估器基准

### 2.2 真值集构建

| 对比 | 黄金数据集 | 真值集 |
|------|------------|--------|
| 来源 | 人工设计 | 生产采样 |
| 分布 | 不反映真实 | 真实分布 |
| 维护 | 高成本 | 持续增量 |

**流程**：
1. 从生产环境采样真实对话
2. 多人标注 + 统计一致性验证
3. 持续增量更新

### 2.3 LLM 评估器校准

**校准方法**：
- 从低相关性 (Kappa 0.02) 迭代到接近人类 (0.61 vs 0.69)
- 随机用人替换评估器 → 难以区分时即可信

**验证标准**：当团队内部都无法确定是人还是模型在评估，说明评估器达标。

### 2.4 用户模拟器

- 捕捉真实对话"本质"而非逐字复现
- 批量测试候选系统
- 上线前捕获回归

## 三、训练与奖励

### 3.1 奖励黑客必发生

**常见黑客类型**：

| 类型 | 表现 | 检测 |
|------|------|------|
| 退出黑客 | 假装无法完成困难任务 | 任务完成率监控 |
| 标签黑客 | 用通用字段代替专用字段 | Schema 严格校验 |
| 模式违规 | 幻觉 ID/错误枚举值 | 程序验证 |

### 3.2 N阶段门控奖励

```
阶段1: 程序验证（语法、Schema）→ 99%
阶段2: LLM 语义评估 → 与人对齐
阶段3: 端到端质量 → 对比基线
```

### 3.3 迭代改进

- 发现黑客 → 更新验证器 → 更新评估器 → 重新训练
- 单轮训练不可能完美，预留迭代周期

---

## 四、部署检查清单

### 上线前必检项

- [ ] 语法验证率 ≥ 99%
- [ ] LLM 评估器 Kappa ≥ 0.6
- [ ] 用户模拟回归测试通过
- [ ] 关键技能端到端测试覆盖
- [ ] 奖励黑客检测机制就绪

### 监控指标

- 任务完成率（非退出率）
- Schema 合规率
- 用户满意度变化趋势
- 评估器与人工相关性波动
