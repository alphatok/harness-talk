# Agent 系统生产构建实践建议

> 基于 Shopify Sidekick (ICML 2025) 核心经验

---

## 一、架构设计

### 1. 从简单的 Agent 循环开始

- 单模型 + 工具集 + 环境交互，足以解决大部分问题
- **不要过早跳入多 Agent 架构** — 延迟爆炸，调试地狱
- 让模型自行决定处理方式，不要预制工作流

### 2. 工具策略：底层原语 > 任务函数

| 错误做法 | 正确做法 |
|---------|---------|
| 为每个场景建工具（get_weather、get_stock、create_discount...） | 建底层原语（读数据、写数据、查文档） |
| 工具数量 50+，边界模糊 | 核心工具 5-10 个，通用性强 |
| 修改一个工具影响全局 | 修改只影响该工具调用时机 |

### 3. 用"即时指令"替代系统提示膨胀

- 指令随工具结果返回，不堆在 system prompt 里
- 好处：system prompt 保持精简、缓存友好、低影响范围
- 适合放 system prompt：人格、回复风格、国际化等全局行为
- 适合放工具响应：边缘案例、引文格式、特定工具的约束

### 4. Agent 交接谨慎使用

- 主 Agent 把复杂子任务交给专门子 Agent
- 子 Agent 必须把回复权交还给主 Agent（保持统一输出风格）
- 只用于后台任务，不要在实时聊天中做

## 二、评估体系

### 1. 抛弃"凭感觉测试"

- 感觉良好就发布 → 问题：无法判断改进、难以调试、回滚频繁
- 必须建立系统化离线评估

### 2. 评估三要素

```
用户模拟器 → 候选系统 → LM 评判员
```

- **用户模拟器**：LM 模拟真实用户意图，与候选系统对话
- **候选系统**：一次只改一个变量（新提示 / 新工具 / 新配置）
- **LM 评判员**：读完整个对话，按多维度评分

### 3. 构建"真实情况集"（即规格）

- ~1000+ 条对话，持续增长
- 覆盖生产分布 + 所有已知边缘案例
- PM + 专家 3-5 人标注评分
- 5-7 个评估维度：目标完成度、安全、情绪、事实依据等
- 统计一致性目标：Kendall's Tau / Cohen's Kappa ~0.7

### 4. 回放仿真 + AA 验证

- 从生产取真实对话 → 提取意图 → 用候选系统重放
- AA 测试：同样系统重放 → 验证评判员评分与生产一致
- 通过后，离线跑 AB 对比 → 有把握再上线

### 5. 评判员持续进化

- 版本 0 可能只有 0.02 相关性 → 迭代到 0.11 → 持续提升
- 每次生产发现新边缘案例 → 加入真实情况集 → 重新调整评判员提示
- 评判员一开始就要强（不像 GAN 需要逐步训练）

---

## 三、RL / GRPO 训练

- 利用已验证的用户模拟器 + 评判员构建 RLHF 流水线
- 真实情况集随生产分布演进 → 评判员提示同步调整
- 核心转变：从"改进模型"变成"持续改进评判员 + 评估流程"

---

## 一句话总结

| 层面 | 底线 |
|------|------|
| 工具 | 底层原语 + 即时指令，拒绝工具膨胀 |
| 评估 | LM 模拟器 + LM 评判员离线跑通，再上线 |
| 数据 | 真实情况集是活规格，持续生长，持续对齐 |
