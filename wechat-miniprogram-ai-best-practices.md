# 微信小程序 AI SKILL 最佳实践建议

> 基于微信小程序 AI 官方最佳实践提炼，聚焦 Agent 系统设计与实现

---

## 1. 核心定义

一句话：**"信息源放对位置，比写什么更重要"** — 同一份指令写在不同的信息源（mcp.json description / content / SKILL.md），准确率天差地别，权重高的位置（如 content）写错话术会直接覆盖低权重位置（如 SKILL.md）的约定。

---

## 2. 信息源权重体系

| 层级 | 信息源 | 权重 | 应该写 | 不该写 |
|------|--------|------|--------|--------|
| **L1** | content | ★★★★★ | 本次结果 + 下一步动作 | 接口功能描述、业务编排规则 |
| **L2** | mcp.json description | ★★★★ | 接口功能、调用时机、不适用场景 | 跨接口规则 |
| **L2** | inputSchema.description | ★★★★ | 参数语义、取值来源、缺省处理 | 业务流程 |
| **L3** | SKILL.md | ★★★ | 业务流程编排、跨接口约束 | 接口本身功能描述 |

**落地检查**：每次写指令前先问"这条指令属于哪一层？放对层了吗？"

---

## 3. mcp.json 声明规范

### 3.1 接口 description

```
首句格式：业务对象 + 核心操作
推荐："搜索饮品。按关键词、温度、杯型检索……"
反例："按关键词、温度、杯型搜索商品列表（仅饮品）……"
```

**关键点**：
- 接口名语义化（`searchDrinks` 优于 `search`）
- 不写内部实现细节
- 同名字段跨接口统一命名

### 3.2 字段 description

```
普通字段规则：
- 举例 ≥ 2 个不同样本（防被当默认值）
- 配明确缺省处理

ID 字段规则：
- 显式声明取值来源接口
- 禁止从自然语言推断 / 编造
- 上下文无可用 ID 时，先调上游接口
```

---

## 4. content 返回：事实 + 动作两段式

```
反例：{content: "接下来请务必为用户展示订单确认卡片。"}
推荐：{content: "已根据所选规格生成订单。请展示确认卡片，引导用户核对下单。"}
```

**失败/空结果三要素**（必须同时出现）：
1. 陈述具体原因
2. 给出下一步出口
3. 指出不应做的动作（配对联出，禁止只写禁令不给出口）

---

## 5. SKILL.md 只写"整个业务怎么跑"

- 节点名与 mcp.json 完全一致（含大小写）
- 从用户意图分支，每分支对应典型表达
- 标注用户操作节点（区分谁触发下一步）
- 依赖关系表按接口维度列前置条件
- 业务约束类型：输出形态 / 执行顺序 / 并发串行 / 数据来源

---

## 6. 通用写作铁律

| 规则 | 说明 |
|------|------|
| **一处写** | 同约束只写一处，重复易冲突 |
| **高权重优先** | 硬约束放权重高的位置 |
| **禁令配出口** | 每条"不要做 X"都要配"应做 Y" |

---

## 7. 上行消息文案原则

- 用户视角 + 第一人称 + 生活化口语
- 异常也要转成用户操作表达（不上行系统消息）
- 仅限用户已明确表达的信息
- 敏感信息脱敏，不用指代模糊词

---

## 8. 落地检查清单

- [ ] 每条指令确认所属信息源层级
- [ ] content 使用"事实 + 动作"两段式
- [ ] 失败分支同时给出原因、出口、禁令
- [ ] mcp.json 字段说明配多举例 + 缺省处理
- [ ] SKILL.md 节点名与 mcp.json 严格一致
- [ ] 同约束只写一处，不重复
