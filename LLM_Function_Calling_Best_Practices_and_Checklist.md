
# 🏆 LLM Function Call 实战避坑与生产上线 60 条硬核军规

## 一、 设计与认知篇：反直觉陷阱与避坑指南 (20条)

1. **参数名比描述更重要**：LLM 对“语义化键名”极度敏感，命名为 `user_email` 比命名为 `a` 加上 100 字描述效果更好。
2. **长描述是毒药**：描述超过 50 个词会稀释模型的 Attention 权重，将描述压缩到 20 字以内的“动词+宾语”结构准确率最高。
3. **不存在 `optional` 关键字**：JSON Schema 标准中没有 `optional`。想让参数非必填，从 `required` 数组中剔除即可，大模型会自动理解。
4. **警惕自由文本（String）**：String 是参数解析失败的万恶之源。只要取值范围有限，坚决使用 `Enum`（枚举），比如用 `["zh", "en"]` 替代 `语言代码`。
5. **不要让大模型做数学题**：如果业务需要 `当前年份 - 1`，让模型提取“当前年份”，然后在你的本地代码里做减法，绝不要让模型在参数里自己算。
6. **工具数量的“15个”死亡阈值**：单次注入超过 15 个工具会导致模型随机乱选。工具过多时，必须先通过 RAG 或意图识别过滤，只给模型注入 Top-5 相关工具。
7. **“上帝工具”必死**：把查询、修改、删除全塞进一个 `manage_user` 工具中会让模型彻底混乱。遵循单一职责原则（SRP），拆分为读写独立的细分工具。
8. **并行调用的依赖地狱**：模型不懂时序。如果操作 B 依赖操作 A 的结果，必须在提示词中强硬规定“严禁将 A 与 B 并行执行”。
9. **嵌套过深的 JSON 是灾难**：参数结构超过 2 层嵌套（如对象里套数组再套对象），括号匹配错误率直线上升。尽可能扁平化参数。
10. **大模型是“懒惰的复读机”**：不加约束的话，模型拿到工具的 JSON 结果会直接甩给用户。必须在系统提示词中规定：“必须将工具数据转化为人类自然语言回复”。
11. **真假 Boolean 陷阱**：在某些弱模型中，让它输出 `true/false` 容易出错，改用 `Enum: ["yes", "no"]` 反而能大幅提升推理稳定性。
12. **别让大模型猜时间**：模型不知道“今天”是几号。必须在 System Prompt 中注入系统当前时间（包含时区），否则与时间相关的工具调用100%出错。
13. **工具描述避免使用“可以用来...”**：弱描述：“这个工具可以用来查天气”。强描述：“获取指定城市的当前温度和湿度”。直接使用祈使句。
14. **One-Shot 示例胜过千言万语**：在复杂参数的 description 中，直接写一个例子（如 `例如："2023-10-01"`），比解释格式规则管用得多。
15. **工具 Schema 会吃掉大量 Token**：每次对话，所有的工具定义都会作为输入 Token 消耗。定期精简不需要的字段，能省下巨额 API 费用。
16. **小心“幻觉参数”**：模型不仅会捏造工具名，还会给你传入 Schema 中根本没有定义的参数。后端严禁直接 `**kwargs` 照单全收。
17. **不要过度信任 `tool_choice="auto"`**：在需要强制收集用户表单的场景，直接指定 `tool_choice={"name": "submit_form"}` 比让模型自己猜有效得多。
18. **缺乏常识的默认值**：如果参数 A 是必填的，但用户没提供，大模型可能会自己瞎编一个（比如缺省名字填入"张三"）。必须指定：“若用户未提供，必须先向用户提问，禁止自行编造”。
19. **读写权限未隔离**：不要给具有“读取任意网页”工具的 Agent 赋予“执行高危 SQL”的工具，外部网页的提示词注入会让 Agent 执行恶意写操作。
20. **忽视工具的边界感**：当模型问用户“你需要查天气还是订票”时，用户回答“查天气”，大模型可能同时调用两个工具。明确告知模型“单次请求最多解决一个核心意图”。

## 二、 开发与异常处理篇：反直觉陷阱与避坑指南 (20条)

21. **缺失死循环拦截 (Max Loops)**：模型调用工具 -> 报错 -> 模型再次重试 -> 再次报错。如果没有全局的 `max_turns` 强制中断（建议设为 3-5），你的破产就在今夜。
22. **返回给模型的上下文污染**：不要把接口返回的 1MB 原始 JSON 塞回给模型，这会撑爆 Context 并让它忘记原始问题。必须在代码侧提取出核心字段再返回。
23. **吞没异常是死罪**：工具执行报错时，绝不要在后端直接 `sys.exit()` 或返回空字符串。必须将 `{"error": "错误详情"}` 作为 Tool Message 喂回给模型，让其自我纠正。
24. **忽略类型漂移**：Schema 里定义的是 `Integer`，模型真传过来时可能是 String `"123"`。代码侧提取参数时必须做强制类型转换。
25. **破坏对话历史结构**：模型发起 Tool Call 后，必须依次追加 Assistant Message (含 tool_calls) -> Tool Message (含执行结果)，漏掉任何一个都会导致下一轮请求报错。
26. **忘记校验 `tool_call_id`**：返回 Tool 结果时，`tool_call_id` 必须与模型发起调用的 ID 完全一致，否则 API 会直接报 400 错误。
27. **不要盲信 `json.loads`**：模型生成的参数偶尔会带有 Markdown 代码块包裹，或者有转义错误。必须实现一段健壮的 JSON 清洗与安全解析逻辑。
28. **流式输出的解析地狱**：Streaming 模式下的 Function Call，返回的是 JSON 的碎片（chunks）。必须在本地维护一个缓冲区拼接完整后再执行，不要半途抛出异常。
29. **忽略并发调用冲突**：模型并行调用多个工具时，如果你在代码里使用同步阻塞执行，会严重拖慢响应。必须使用 `asyncio.gather` 并发执行所有的 Tool Calls。
30. **误把前端交互当做工具调用**：如果只是想在前端展示一个日历组件，不要走完整的 Function Call 链路，这太重了。直接让模型输出特定的 XML/Markdown 标签由前端渲染即可。
31. **模型切换带来的 Schema 崩溃**：从 GPT-4 切换到国产模型时，国产模型可能不支持复杂的嵌套 Schema 或特定关键字。做好兼容性降级测试。
32. **强求模型处理鉴权**：绝不要让模型在参数里提取用户的 Token 或 Password。鉴权必须在你的后端拦截器中通过 Session 隐式完成。
33. **忽略工具的幂等性**：大模型可能会因为网络重试，对同一个订单调用两次 `pay_order`。写操作工具必须通过全局的请求 ID 实现强幂等。
34. **不处理空参数调用**：有些函数不需要参数（如 `get_current_location()`），但部分模型会强行塞入一个 `{}` 甚至是 `{"dummy": "data"}`。代码侧要包容这种冗余。
35. **工具名大小写地狱**：Schema 里是 `getWeather`，模型可能因为看了某篇 Python 教程而返回 `get_weather`。你的注册表最好实现大小写、下划线脱敏匹配。
36. **丢弃最终结论**：很多新手在执行完工具后，直接把工具的结果发给前端。正确的做法是：工具结果必须发回给大模型，让大模型总结出自然语言后再发给前端。
37. **不支持优雅降级**：当调用的外部 API 宕机时，应该返回“系统当前不可用，请安抚用户”，而不是让模型看到 HTTP 500 堆栈信息然后胡言乱语。
38. **过度复杂的参数校验**：与其在 prompt 里写几百字教模型怎么填复杂的正则表达式，不如让它传原词，由你的后端代码用正则提取并纠正。
39. **忽视 `finish_reason`**：必须严格判断响应体的 `finish_reason` 是 `tool_calls` 还是 `stop`。依靠解析文本内容来判断是否调用了工具是极不稳定的。
40. **忘记开启 Strict Mode**：如果使用的是 OpenAI 较新模型，务必在 Schema 顶层加上 `"strict": true`，这能物理消灭 99% 的 JSON 格式错乱问题。

## 三、 生产级上线 Checklist (20条)

### 🛡️ 安全与风控
41. [ ] **身份隔离鉴权**：工具代码中是否自动读取了后端上下文的用户 ID？（严禁从 LLM 的传参中获取 `user_id` 进行越权数据操作）。
42. [ ] **人类确认机制 (HITL)**：涉及资金、删除、公开群发等不可逆的危险函数，是否在前端拦截并强制要求用户点击“确认执行”按钮？
43. [ ] **输入防注入清洗**：对于要存入数据库或渲染到页面的工具输出结果，是否做好了 XSS 和 SQL 注入的实体转义与过滤？
44. [ ] **敏感信息脱敏**：工具返回给 LLM 的日志或数据中，是否已经将密码、身份证号、银行卡号等 PII（个人身份信息）做了 `***` 掩码处理？
45. [ ] **白名单参数校验**：即使 LLM 传来了合法的 Enum 字符串，后端是否仍进行了二次校验，防止接口被恶意用户通过代理直接抓包篡改调用？

### ⚙️ 稳定性与性能
46. [ ] **硬性超时熔断**：所有外部 API 调用是否都设置了极其严格的超时时间（如 3-5 秒）？
47. [ ] **防无限循环锁**：是否在对话上下文循环体中设置了 `MAX_TOOL_CALL_LOOPS = 5` 的硬性跳出条件？
48. [ ] **返回数据截断**：工具返回给大模型的文本内容是否做了硬截断（例如 `result[:2000] + "...(truncated)"`），以防止 Token 耗尽或 OOM？
49. [ ] **连接池防击穿**：并行 Tool Call 会造成并发尖峰，数据库或 Redis 连接是否使用了连接池并设置了合理的 Max Connections？
50. [ ] **API 频率限制**：是否针对单个用户设置了 Function Call 的限流（Rate Limit），防止恶意用户刷量消耗你的 Token 余额？

### 🔍 可观测性与调试
51. [ ] **工具调用全链路日志**：是否打印了结构化日志：`TraceID | 请求意图 | 调用的工具名 | 入参JSON | 耗时 | 响应状态`？
52. [ ] **幻觉率监控告警**：是否针对“调用不存在的函数名”和“缺少必需参数”这两种典型幻觉做了指标埋点和高频报警？
53. [ ] **Token 消耗分离统计**：是否能在账单/仪表盘中，清晰区分出纯文本消耗的 Token 和 Tool Schema 消耗的 Token？
54. [ ] **调试溯源机制**：在测试环境，前端是否能展开一个 Debug 面板，查看到底触发了哪几个函数以及原始 JSON 参数？
55. [ ] **失败归档**：执行失败的 Function Call（尤其是参数解析失败），是否被自动转储到了专用的 Review 数据库中供人工调优？

### 🧩 业务逻辑与兼容性
56. [ ] **向前兼容机制**：当修改工具参数结构时，老版本客户端缓存的会话能否正常降级，而不引起模型严重幻觉？
57. [ ] **空结果处理**：当数据库查询为空时，返回给模型的是否是明确的 `"msg": "未找到相关数据"`，而不是 `null` 或 `[]`（这容易引发模型瞎编）？
58. [ ] **日期和时区强关联**：依赖日期的核心业务，是否在本次会话的顶部注入了用户设备当前的准确时区和系统时间？
59. [ ] **并行关闭开关**：如果是使用的 GPT-4o 等模型，对于极其敏感的时序操作，是否设置了 `parallel_tool_calls=false` 的降级开关？
60. [ ] **回滚预案**：当大模型厂商突然更新模型权重导致 Function Call 准确率暴跌时，是否有秒级切换至备用模型（如从 GPT-4o 瞬间切到 Claude 3.5）的配置开关？
